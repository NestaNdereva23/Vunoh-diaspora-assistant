## Overview
 
Kenyans in the diaspora currently rely on informal channels  WhatsApp messages to relatives, phone calls, word of mouth  to manage tasks back home. These channels are slow, unreliable, and leave no audit trail when things go wrong.
 
This application gives diaspora customers a single interface to:
- Submit a request in plain English
- Get an AI-extracted, structured task with a unique reference code
- Receive confirmation across three channels: WhatsApp, Email, and SMS
- Track task progress through a live dashboard

## Data Models
 
### `Task`
The core record created for every customer request. Stores the original message, AI-extracted intent and entities, calculated risk score, current status, and assigned team.
 
Key fields:
- `task_code` short human-readable reference (e.g. `VNH-A3F9`) generated on first save
- `intent` one of: `send_money`, `hire_service`, `verify_document`, `airport_transfer`, `check_status`
- `entities`  JSON field holding AI-extracted details (amount, recipient, location, urgency, document type, etc.)
- `risk_score`  integer 0–100 calculated by the local rules engine
- `status`  `Pending` → `In Progress` → `Completed`
- `assigned_team`  `Finance`, `Operations`, `Legal`, or `Logistics`
### `TaskStep`
An ordered checklist of fulfilment steps generated by the AI for each task. Makes it possible to mark individual steps complete without touching the others.
 
Key fields:
- `step_order`  integer starting at 1, enforces correct display sequence
- `description`  plain-English step description
- `is_complete`  boolean, flipped as the assigned team works through the task
- `unique_together (task, step_order)`  no duplicate step numbers per task
### `Message`
Stores all three confirmation messages per task WhatsApp, Email, and SMS as individual rows. Written atomically alongside the task so all three either save or none do.
 
Key fields:
- `channel` - `whatsapp`, `email`, or `sms`
- `body` - the full message text
- `unique_together (task, channel)` - prevents accidental duplicate messages per channel
### `StatusHistory`
An append-only audit log. Every time a task's status changes, a new row is inserted  existing rows are never updated or deleted. 
 
Key fields:
- `from_status` - what the status was before the change
- `to_status` - what it changed to
- `changed_at` - timestamp of the change
- `note` - optional context.

### How I Designed the System Prompts

The application makes three separate LLM calls, each with its own focused prompt: intent extraction, step generation, and message generation. This was a deliberate choice over one large combined prompt.

**What I included in every prompt:**
- A clear role definition at the top -> *"You are a fulfilment coordinator at Vunoh Global"* to anchor the model in the right context before any instructions
- An explicit instruction to return only valid JSON, explanation, or markdown fences. This is the most important line in each prompt. Without it the model wraps output in triple backticks or adds an introductory sentence, both of which break `json.loads()`
- Worked examples showing exact input and expected output. The intent extraction prompt has three full examples covering `send_money`, `verify_document`, and `hire_service`. Examples outperform instructions alone for consistent JSON structure
- Explicit null handling -> *"if a field is not mentioned, set it to null, do not guess"*. Without this the model invents plausible-sounding values for missing fields, which is worse than an empty field for a financial platform

**What I deliberately excluded:**
- Long lists of edge cases and exceptions. Over-specified prompts cause the model to fixate on the constraints rather than the task. Each prompt covers the main cases and trusts the model on the rest
- Any instruction to explain its reasoning. Chain-of-thought is useful for complex reasoning tasks but here it produces output that is harder to parse. The goal is clean JSON, not a reasoned answer
- Combined prompts. Asking for intent, steps, and messages in one call would produce a large nested JSON object. The larger and more complex the expected output, the more likely the model is to drop a field or mis-nest something silently
---

### One Decision Where I Overrode the AI

When Claude first suggested the `services.py` architecture, it proposed a single `process_task()` function that made one combined LLM call returning intent, steps, and all three messages in a single large JSON object. The argument was fewer API calls, lower latency, simpler code.

I overrode this and split it into three separate calls.

My reasoning: this platform handles real money transfers and legal document verification on behalf of people who are thousands of kilometres from home. If the combined call fails partway through or returns a malformed response, you lose everything intent, steps, and messages and have nothing to save to the database. With three separate calls, if message generation fails the task is still created with correct intent and steps. The failure is isolated, logged, and recoverable. Reliability is worth more than saving one API round trip for a task management platform.

### One Thing That Did Not Work as Expected

The risk scoring engine implementation I was initially using was scoring low on important user messages that should realistically have been treated as high risk. The first version relied on flat additive scoring, where each factor such as urgency, amount, or recipient verification was evaluated independently. 

This created situations where combinations of risky behavior were not being weighted strongly enough. For example, an urgent request involving a large amount and an unverified recipient could still end up with a moderate score because the engine was not considering how those signals interact together in real-world fraud scenarios.

**How I resolved it:**

I refactored the scoring engine to use contextual and compound risk evaluation instead of only independent scoring rules. I introduced:
- weighted intent categories
- safer and more flexible amount parsing
- interaction-based risk rules (e.g. high urgency + large transaction amount)
- normalized scoring between 0–100
- human-readable explanations for why a score was assigned

I also added trust reduction logic for returning customers with clean histories to better reflect realistic operational behavior. This made the scoring system more explainable, easier to maintain, and more aligned with the Kenyan diaspora context described in the assignment.
